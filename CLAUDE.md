This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run build              # vite build (frontend) -> ../dist, served by the Worker's assets binding
npm run dev                 # wrangler dev — single process serving API + DO + built frontend on :8787
npm test                    # full suite: jsdom/pure-fn tests, then Workers-runtime tests
npm run test:unit           # vitest --config vitest.config.ts (frontend/src + src/shared, jsdom)
npm run test:workers        # vitest --config vitest.workers.config.ts (src/, real D1/DO via vitest-pool-workers)
npx tsc --noEmit             # typecheck the whole repo (frontend + src share one tsconfig)
npm run db:migrate:local     # applies src/db/schema.sql to the local D1 instance
npm run db:migrate:local:0001 / :0002   # applies one migration file locally (see src/db/migrations/)
npm run db:migrate:prod:0001            # applies one migration file to the remote prod D1 (there's no :prod:0002 script yet — run migration 0003 by hand with `wrangler d1 execute fives --remote --env production --file=...` too)
npm run deploy                # build + wrangler deploy --env production (uses the `env.production` block in wrangler.jsonc)
```

Run a single test file: `npx vitest run --config vitest.config.ts path/to/file.test.tsx` (or swap in `vitest.workers.config.ts` for anything under `src/`). There is no separate lint script.

For active frontend development with hot reload, run `wrangler dev` (port 8787) and `cd frontend && npx vite` (port 5173) side by side — `frontend/vite.config.ts` proxies `/api` and `/ws` from Vite's dev server to the Worker. This is a dev-only convenience; in the built/deployed app there is exactly one Worker serving everything (API, WebSocket upgrades, and the static SPA) from one origin.

## Architecture

**Fives** is a 2-captain live bidding draft for 5-a-side soccer squads: two captains draft 5 players each from a shared pool of exactly 10, via alternating turn-based bidding, on Cloudflare Workers + Durable Objects + D1. There are no user accounts — captain identity is a bearer token embedded in a join link.

### Two test configs, two runtimes
`vitest.config.ts` (jsdom + `@testing-library/react`) covers `frontend/src/**` and `src/shared/**` — pure functions and presentational/hook code with mocked WebSocket/fetch. `vitest.workers.config.ts` (`@cloudflare/vitest-pool-workers`) covers everything else under `src/` and runs against **real** D1 and Durable Object bindings defined in `wrangler.jsonc` — these are integration tests, not mocks. When adding a test, put it in the config that matches what it touches; `src/shared/*.test.ts` is deliberately excluded from the workers config.

### Request routing (`src/index.ts`)
A single Hono app mounts four independent sub-routers: `playersRouter` (`/api/players`), `playersAdminRouter` (`/api/admin/players`, `src/routes/players-admin.ts`), `gamesRouter` (`/api/games`, now the largest router — see below), and `wsRouter` (`/ws/games`, `src/routes/ws.ts`). Each router is self-contained and independently testable via `router.request(...)` without booting the full app — keep new route groups structured this way rather than adding routes directly to `index.ts`.

Two extra routes are registered directly on the top-level app, ahead of the final `app.get("*", ...)` static-asset fallthrough: `GET /game/:gameId` and `GET /showcase/:slug`. Both fetch the built `index.html` from `ASSETS`, and — only when the game is `completed` (for `/game/:id`) or published (for `/showcase/:slug`) — rewrite it with `injectOgMeta` (`src/lib/og.ts`) before serving, so a link-preview crawler (Slack/iMessage/Twitter) unfurls the actual squad share image instead of the generic app icon. Everything else still falls through to `env.ASSETS.fetch()` (Workers static assets, SPA fallback).

The default export also implements a `scheduled` handler (cron `*/15 * * * *`, see `wrangler.jsonc`'s `triggers`) that runs three sweeps every 15 minutes: `deleteExpiredGames`, `deleteExpiredPublicPosts` (both `src/lib/cleanup.ts`), and `sendVotingClosedNotifications` (`src/lib/notify.ts`).

The `Env` bindings (`src/index.ts`) are: `DB` (D1), `GAME_ROOM` (DO namespace), `ASSETS`, `PLAYER_IMAGES` (R2 bucket for roster player photos), `CREATE_GAME_RATE_LIMITER` / `VOTE_RATE_LIMITER` / `COMMENT_RATE_LIMITER` (Workers rate limiting; 5/60s, 10/60s, 10/60s per client IP respectively — enforced in `routes/games.ts`'s `POST /`, `POST /public/:slug/vote`, and `POST /public/:slug/comments`), `THESPORTSDB_API_KEY` (optional, used by `players-admin` to import league/team/player data from TheSportsDB), `RESEND_API_KEY` / `RESEND_FROM_ADDRESS` (optional; `notify.ts` no-ops if either is unset), and `APP_BASE_URL`. Note `wrangler.jsonc`'s `env.production` block currently omits `COMMENT_RATE_LIMITER` from its `ratelimits` array — check before assuming prod parity with local/default config.

### `GameRoom` Durable Object (`src/durable-objects/game-room.ts`) — the authoritative game engine
One DO instance per game (keyed by `idFromName(gameId)`). Core game logic lives in plain async methods (`init`, `handleCaptainConnected`, `proposeNextPlayer`, `placeBid`, `pass`, `getState`, `getCaptainForToken`) that return an `ActionResult<GameState> = {ok:true,state} | {ok:false,code,message}` rather than throwing — throwing custom errors across the DO RPC boundary was found to crash `vitest-pool-workers`' isolated-storage bookkeeping, so this return-based pattern is load-bearing, not stylistic. More methods follow the same pattern for auxiliary features: `setCaptainName(captain, name)` (display-name, sanitized + profanity-checked), `sendChatMessage(captain, text)` (in-game chat, capped at `MAX_CHAT_HISTORY` = 200 entries, also sanitized/profanity-checked), and `requestPublish(captain, notifyEmail)` (records one captain's consent to publish the completed game to the public voting page; once *both* captains have consented, allocates a `public_slug` — a separate slug namespace from game ids, same collision-retry pattern — and writes `published_at`/`public_slug`/`voting_closes_at`/notify-emails through to D1 in one shot). All of these are callable directly (cross-file RPC from `routes/games.ts`, or via `runInDurableObject`/direct stub calls in tests) and back the thin WebSocket layer (`fetch`/`webSocketMessage`/`webSocketClose`, using the Hibernation API via `ctx.acceptWebSocket`). The DO also registers a Hibernation-API auto-response (`ctx.setWebSocketAutoResponse`) so ping/pong heartbeats are answered without waking the DO.

State is persisted as a single JSON blob in the DO's own SQLite storage (`ctx.storage.sql`), written synchronously on every mutation with no in-memory-only cache — every read goes back to storage. This means a hibernation evict/reload cycle can't lose state, at the cost of re-parsing JSON per call (acceptable at this scale).

Every mutation also renews a DO alarm to `Date.now() + GAME_DO_TTL_MS` (2 days). If no mutation happens before it fires, `alarm()` wipes the DO's stored state — an idle/abandoned game's *DO* state is dropped independently of the `deleteExpiredGames` D1 sweep (`src/lib/cleanup.ts`), which purges the corresponding `games`/`game_pool` rows on a longer (7-day) timer. Don't assume the two are the same mechanism.

Publishing is a two-consent handshake, not a route: the frontend sends `request_publish` (optionally with a `notifyEmail`) over the game's own WebSocket once the game is `completed`; the public showcase/vote/comments routes in `routes/games.ts` read `published_at`/`public_slug` from **D1**, not from the DO, so once both captains have consented the DO's involvement is done.

**Critical protocol detail**: every mutating WebSocket message handler (`propose_next_player`, `place_bid`, `pass`) broadcasts its specific event (`round_started`, `bid_placed`, `round_settled`, `game_completed`) **and then** broadcasts a fresh `state_snapshot`. The frontend's `useGameSocket` hook only updates displayed state on `state_snapshot` — it intentionally does not hand-parse the granular events into partial state updates, relying on the server being authoritative. If you add a new mutating message type, it must end with a `state_snapshot` broadcast or the UI will silently stop updating. Non-game-state events (`captain_joined`, `game_started`, `chat_message`, and `chat_history` sent once on connect) don't carry state and aren't subject to this rule.

### Shared contracts (`src/shared/`) — the interface both sides code against
- `constants.ts`: `STARTING_BUDGET` (250,000,000), `MIN_BID_INCREMENT` (5,000,000), `POOL_SIZE` (10), `SQUAD_SIZE` (5), `GOALIES_IN_POOL` (2 — an *exact* count per pool, not just a floor, despite the name reading like a minimum), plus field-length caps (`MAX_CAPTAIN_NAME_LENGTH`, `MAX_CHAT_MESSAGE_LENGTH`, `MAX_PLAYER_NAME_LENGTH`, `MAX_LEAGUE_NAME_LENGTH`, `MAX_CLUB_NAME_LENGTH`, `MAX_NATION_NAME_LENGTH`, `MAX_IMAGE_URL_LENGTH`, `MAX_EMAIL_LENGTH`, `MAX_COMMENT_TEXT_LENGTH`, `MAX_COMMENT_AUTHOR_NAME_LENGTH`), roster pagination sizes (`ROSTER_PAGE_SIZE`, `MAX_ROSTER_PAGE_SIZE`), and retention/window constants for the publish-voting feature: `GAME_DO_TTL_MS` (2 days), `ABANDONED_GAME_RETENTION_MS` (7 days), `PUBLISH_VOTING_WINDOW_MS` (2 hours), `PUBLIC_POST_RETENTION_MS` (7 days), `PUBLIC_FEED_SIZE` (50), `PUBLIC_COMMENTS_LIMIT` (200).
- `types.ts` / `protocol.ts`: `GameState`, `RoundState`, and the full `ClientMessage`/`ServerMessage` discriminated unions — including `join`, `send_chat`, and `request_publish` client messages, and `captain_joined`/`game_started`/`chat_message`/`chat_history` server messages alongside the core game-state ones. `GameState` also carries `publishConsent: Record<Captain, boolean>`. `Player` also carries `league`, `externalId`, and `archivedAt`.
- `rules.ts`: `computeReserve`/`computeMaxLegalBid`/`isLegalBid` — the reserve-rule math (a captain can never bid more than `budget - (slots still needed after this player) × minIncrement`). This is imported by both the DO (authoritative enforcement) and the frontend (`BidControls`, for instant UX clamping only — the server is what actually enforces it).
- `sanitize.ts` (`sanitizeText`) and `moderation.ts` (`containsProfanity`): shared text-cleaning used for captain names, chat messages, and (via the same pattern re-implemented in `routes/games.ts`) public showcase comments/usernames.
- `formation.ts`: `layoutFormation`, pure pitch-position math consumed by `src/lib/squad-image.ts` to place players on the rendered share image.

Import from `src/shared/` rather than redefining these types locally; frontend files reach it via relative paths like `../../../src/shared/...` since `frontend/` and `src/` are separate root-ish trees under one tsconfig.

### Game rules encoded in the state machine
Round order is fully randomized once at game creation and fixed as `proposalOrder` on each pool entry. The "first bidder" for each round alternates every round (round 1's first bidder is a one-time coin flip stored at creation). The first bidder **must** place an opening bid — passing is only legal in the `awaiting_response` subphase, which structurally rules out "pass with nothing on the table" rather than special-casing it. A pass settles the round: the player goes to whoever holds `currentBid`, at that price.

### Game creation (`routes/games.ts`)
`POST /api/games` builds the 10-player pool one of two ways: a **random draw**, optionally filtered by `leagues`/`clubs`/`nations`, or a captain **hand-picked pool** via `selectedPlayerIds` (exactly `POOL_SIZE` unique ids). Both paths enforce `GOALIES_IN_POOL`. Game ids are human-readable two-word slugs (`src/lib/slug.ts`'s `generateGameSlug`, e.g. `swift-otter`) generated with a collision-retry loop against D1, falling back to a UUID after `MAX_SLUG_ATTEMPTS`. Creation is rate-limited via `CREATE_GAME_RATE_LIMITER` (5/min per client IP).

### Public showcase: voting, comments, share images (`routes/games.ts`, `lib/squad-image.ts`, `lib/notify.ts`, `lib/cleanup.ts`)
Once both captains consent (see the DO's `requestPublish` above), a completed game gets a `public_slug` and becomes visible at `GET /api/games/public` (the feed, newest-first, capped at `PUBLIC_FEED_SIZE`) and `GET /api/games/public/:slug` (one game's squads + vote tallies). Two write endpoints exist for anonymous visitors, both IP-rate-limited and gated on the game not yet being purged:
- `POST /api/games/public/:slug/vote` — body `{ choice: "A"|"B", voterId }`; `voterId` is a random UUID minted once per browser by `frontend/src/lib/voter.ts` and persisted in localStorage, and `game_votes` has a `(game_id, voter_id)` primary key enforced via `INSERT OR IGNORE`, so a repeat vote from the same browser is a silent no-op rather than an error. Voting closes at `voting_closes_at` (`PUBLISH_VOTING_WINDOW_MS` = 2h after publish); voting after that returns 410.
- `POST /api/games/public/:slug/comments` — free-text comments (`game_comments` table), validated with the same sanitize → trim → length-cap → profanity-check pipeline as in-game chat (`parseComment` in `routes/games.ts` deliberately mirrors `sanitizeCaptainName`/chat validation rather than importing it, since a commenter can also explicitly go anonymous, which the captain-name path has no equivalent for).

Share images are PNGs rendered on demand via `src/lib/squad-image.ts` (Satori for layout + `@cf-wasm/resvg` for PNG rasterization, using `shared/formation.ts` for pitch placement) at `GET /api/games/:id/share/{A,B}.png` (solo squad, completed games only), `GET /api/games/:id/share/combined.png`, and `GET /api/games/public/:slug/share/combined.png` (published games). All three check the Cache API (`caches.default`) first and `waitUntil`-populate it with a `max-age=31536000, immutable` response — the satori/resvg dependency chain is loaded via dynamic `import()` inside the handler so it isn't pulled into every cold start, only ones that actually render an image. `src/index.ts`'s `/game/:gameId` and `/showcase/:slug` routes point OG/Twitter meta tags at these same URLs.

Two cleanup/notify jobs run off the cron sweep in `src/index.ts`'s `scheduled` handler (see above): `deleteExpiredPublicPosts` (`lib/cleanup.ts`) tears down a published game's public presence — clearing `published_at`/`public_slug`/`voting_closes_at` and deleting its `game_votes`/`game_comments` — `PUBLIC_POST_RETENTION_MS` (7 days) after publish, while leaving the completed game itself (`status`/`completed_at`/`game_players`) untouched as the permanent historical record. `sendVotingClosedNotifications` (`lib/notify.ts`) emails each captain who opted in with a `notifyEmail` once their game's voting window has closed (via the Resend API, no-op if `RESEND_API_KEY`/`RESEND_FROM_ADDRESS` aren't configured), stamping `voting_closed_notified_at` so a game is never notified twice.

### D1 schema (`src/db/schema.sql` + `src/db/migrations/`)
`players` (soft-delete via `archived_at`, never hard-deleted — historical games must still resolve archived players by id; also carries `external_id` (unique, from TheSportsDB imports), `league`, `image_url`), `games` (now also carrying `voting_closes_at`, `captain_a_notify_email`, `captain_b_notify_email`, `voting_closed_notified_at` from migration `0002`, alongside the earlier `published_at`/`public_slug`), `game_pool` (the 10-player draw + its randomized order for one game), `game_players` (final result, written by the DO when a game completes), `game_votes` (migration `0002`, `(game_id, voter_id)` PK, `choice` CHECK'd to `'A'|'B'`), and `game_comments` (migration `0003`). Numbered migration files under `src/db/migrations/` are additive deltas applied on top of the base `schema.sql` — check there (not just `schema.sql`) for the current live shape of `games`, and add a new numbered file rather than editing `schema.sql` in place for schema changes to an already-deployed table.

### Player images (`routes/players.ts`, R2)
Roster player photos are served from the `PLAYER_IMAGES` R2 bucket at `GET /api/players/images/:key` (strict key format, immutable caching). `players-admin.ts` handles uploads/imports, including pulling player data and images from TheSportsDB by league/team when `THESPORTSDB_API_KEY` is configured.

### Frontend structure
`frontend/src/router.tsx` wires seven routes: `/` (marketing home page), `/roster`, `/games/new`, `/game/:gameId/join`, `/game/:gameId`, `/showcase` (`PublicFeedPage`, the public voting feed), and `/showcase/:slug` (`PublicSquadPage`, one published game's squads + voting + comments), to pages in `frontend/src/pages/`. Captain identity/session is `frontend/src/lib/session.ts` (localStorage, keyed by `gameId`, storing `{token, role, joinUrlForB?, name?}`) — there are no accounts; `POST /api/games` mints both captain tokens upfront, and the join URL embeds captain B's token as `?t=`.

`frontend/src/hooks/useGameSocket.ts` is the sole place that opens/reconnects the game WebSocket. It reconnects indefinitely with exponential backoff (starting at 500ms, doubling up to a 10s cap), runs a client-side heartbeat (ping every 15s when idle, force-closing the socket if no pong arrives within 5s), and resets/re-probes the connection on `visibilitychange` when the tab regains focus. (This replaced an earlier one-shot-retry policy — don't assume the simpler behavior described in old commit messages still applies.)

Two small localStorage-backed helpers support the public/social features, both deliberately client-side only since Fives has no accounts: `frontend/src/lib/voter.ts` (`getVoterId`/`hasVoted`/`markVoted` — one random voter id per browser, reused across every showcase page) and `frontend/src/lib/rivalry.ts` (`getRivalry`/`recordGamePlayedOnce` — a per-opponent-name "games played together" counter; tracks games played rather than a fabricated win/loss record, since a "win" only exists via the optional public vote).

Styling lives in `frontend/src/styles.css` (a single ~1500-line stylesheet, "stadium/matchday" visual theme) — the app is **not** unstyled; earlier notes calling it "bare semantic HTML" are stale. Notable components beyond the core pages: `ChatModal.tsx` (in-game chat UI, paired with the DO's chat feature and `lib/chatToggleSlot.ts`), `CommentsSection.tsx` (public showcase comments, paired with the `/public/:slug/comments` endpoints), `AddPlayerModal.tsx`, `Modal.tsx`, `PlayerCard.tsx`, `BidControls.tsx`, `BudgetPanel.tsx`, `ResultsTable.tsx`, `SquadPanel.tsx`, and `lib/api/{games,players}.ts` for typed fetch wrappers.
